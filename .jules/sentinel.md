## 2024-05-24 - [IDOR in Post Creation and Missing Route Authentication]
**Vulnerability:** The `POST /api/posts` and `POST /api/posts/:id/react` endpoints lacked authentication middleware in the routes definition. Furthermore, the `createPost` controller implicitly trusted `req.body.userId`, allowing any user to potentially create posts masquerading as someone else (IDOR).
**Learning:** Controller logic alone is insufficient if the middleware protecting it is missing from the route. User-supplied identification data (like `userId` in the body) should never be trusted for actions linked to the authenticated user.
**Prevention:** Always verify that routes defining state-changing endpoints (POST, PUT, DELETE) have authentication middleware applied. Within controllers, always rely on `req.user.id` (set by the authentication middleware) to determine the user performing the action, rather than relying on the client to send a `userId`.
## 2025-03-21 - [Unauthorized User Enumeration via Missing Auth on Search Endpoint]
**Vulnerability:** The `/api/users/search` endpoint lacked authentication middleware, allowing any unauthenticated user to enumerate and discover registered users' information (like email addresses and names). This could be exploited for scraping user data. Additionally, the JWT middleware was incorrectly referencing `decoded.userId` instead of `decoded.id`, which could lead to authentication failures, and the controller used `Op.iLike`, which crashed the SQLite test database.
**Learning:** All endpoints that expose user data must be explicitly protected by authentication middleware unless intended to be fully public. Also, middleware token payload keys must match the keys used when signing the token, and database queries should ideally be cross-compatible with the testing database to ensure tests pass.
**Prevention:** Always verify that routes defining endpoints that return sensitive or user-specific data have authentication middleware applied. Ensure consistency between token generation and verification logic. Use standard Sequelize operators (like `Op.like`) instead of database-specific ones (like `Op.iLike`) when possible to maintain test compatibility.
